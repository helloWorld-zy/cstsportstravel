# Feature Specification: 境内跟团游 MVP

**Feature Branch**: `001-domestic-tour-mvp`

**Created**: 2026-06-19

**Status**: Draft

**Input**: 境内跟团游完整交易闭环，旅游预订系统首期 MVP

## Clarifications

### Session 2026-06-19

- Q: PRD中退款审批阈值存在两套定义（500/5000/20000元 vs 1000/5000元），采用哪套？ → A: 采用1000/5000元三级审批方案（≤1000运营审批、1000-5000财务主管审批、>5000总监审批）
- Q: PRD中支付倒计时存在30分钟和15-30分钟两种定义，境内游实际应为多少？ → A: 统一30分钟，所有产品均使用30分钟支付倒计时。此决策覆盖PRD §5.2.1中的15分钟选项，MVP阶段不区分产品类型。后续阶段如需支持差异化倒计时，应将倒计时时长配置为产品级参数。
- F2修复：新增FR-030（管理员MFA），解决宪法Principle III合规缺失
- F3修复：明确30分钟倒计时是MVP阶段的刻意决策，覆盖PRD的15分钟选项

## User Scenarios & Testing

### User Story 1 - 用户注册与登录 (Priority: P1)

消费者通过手机号+验证码注册并登录平台，完成实名认证后即可预订产品。同时支持微信授权快捷登录，降低注册门槛。

**Why this priority**: 用户身份系统是所有交易功能的前置条件，没有登录就无法下单。

**Independent Test**: 新用户可以通过手机号完成注册→登录→实名认证全流程，已有用户可以直接登录。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 用户未注册, **When** 输入手机号并获取验证码, **Then** 系统发送6位数字验证码（5分钟有效），同一手机号60秒内不可重复发送
2. **Given** 用户收到验证码, **When** 输入正确验证码, **Then** 系统创建账号并自动登录，分配默认昵称和头像，返回 Access Token（RS256 签名，15分钟有效期）
3. **Given** 用户已注册, **When** 输入手机号+验证码登录, **Then** 系统验证通过后返回用户信息和 Token
4. **Given** 用户在微信内访问, **When** 通过 OAuth 2.0 授权, **Then** 系统获取微信 OpenID，首次登录自动创建账号并绑定
5. **Given** 用户已登录, **When** 提交姓名+身份证号进行实名认证, **Then** 系统校验通过后标记为"已实名"，可下单境内游产品
6. **Given** 用户连续5次输入错误验证码, **When** 再次尝试, **Then** 系统锁定该手机号15分钟
7. **Given** 用户密码不符合策略, **When** 设置密码, **Then** 系统要求8位以上且包含大小写字母、数字、特殊字符中的至少三种

**前端页面维度：**

8. **Given** 用户访问 Web 端, **When** 打开登录/注册页, **Then** 展示手机号+验证码登录表单和微信授权登录按钮，表单包含手机号输入框、验证码输入框、获取验证码按钮（60秒倒计时）
9. **Given** 用户访问微信小程序, **When** 打开登录页, **Then** 展示 wx.login 快捷登录入口和手机号绑定流程
10. **Given** 用户登录后进入个人中心, **When** 查看实名认证状态, **Then** 展示"未认证"/"已实名"标签，点击可进入实名认证表单（姓名+身份证号输入，实时校验身份证号格式）
11. **Given** 用户在个人中心, **When** 管理常用出游人, **Then** 可添加/编辑/删除出游人（姓名、身份证号、手机号、出生日期、性别），上限20人
12. **Given** 后台管理员访问后台管理系统, **When** 打开登录页, **Then** 展示用户名+密码登录表单

### User Story 2 - 产品搜索与浏览 (Priority: P1)

消费者可以按目的地、出发城市、天数、价格等维度筛选境内游跟团游产品，查看产品详情页中的行程安排、费用说明、退改政策和团期日历。

**Why this priority**: 产品发现是转化漏斗的入口，用户必须能看到产品才能下单。

**Independent Test**: 用户可以通过首页或搜索进入产品列表，使用筛选条件缩小范围，点击进入详情页查看完整信息。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 用户请求产品列表, **When** 传入筛选条件（目的地/出发城市/天数/价格区间）, **Then** 返回符合条件的产品列表，支持分页、排序（综合推荐/价格/满意度/销量/天数）
2. **Given** 用户请求产品详情, **When** 传入产品 ID, **Then** 返回产品完整信息（基础信息、行程、费用说明、退改政策、团期日历、评价统计）
3. **Given** 用户请求团期日历, **When** 传入产品 ID 和日期范围, **Then** 返回每日的成人价、儿童价、库存状态（充足/紧张/售罄）
4. **Given** 产品已售罄, **When** 查询产品列表, **Then** 产品状态标记为"已售罄"，不参与默认排序
5. **Given** 用户输入搜索关键词, **When** 输入过程中, **Then** 返回实时联想推荐（热门目的地→产品名称→景点）

**前端页面维度：**

6. **Given** 用户访问首页, **When** 浏览页面, **Then** 展示搜索框、金刚区导航（境内跟团游/出境跟团游/邮轮游等入口）、Banner 轮播位（3-5秒自动轮播）、热门目的地推荐区（按境内热门分类的图文卡片）、"猜你喜欢"个性化推荐区
7. **Given** 用户进入产品列表页, **When** 浏览列表, **Then** 展示筛选栏（顶部标签：目的地/出发城市/天数/价格 + 侧边抽屉：住宿标准/主题标签/产品等级/交通工具）、排序下拉、产品卡片列表（主图/名称/起价/天数/出发城市/满意度/标签），支持列表模式和网格模式切换
8. **Given** 用户进入产品详情页, **When** 浏览页面, **Then** 展示图片轮播（首屏预加载前3张）、产品经理推荐语、行程详情（按 Day 时间轴展示每日景点/用餐/住宿/交通）、费用包含/不含说明、人群价格区分（成人/儿童/婴儿）、单房差说明、退改政策（不可折叠隐藏）、用户评价（5星制+各维度评分+好评/中评/差评筛选）、团期日历（日历热力图，每日标注成人价/库存状态）、底部固定预订栏（起价+"立即预订"按钮）
9. **Given** 用户在团期日历中查看日期, **When** 浏览未来3个月团期, **Then** 充足日期正常显示、紧张日期橙色标注、售罄日期置灰不可选、超过预订截止日期的团期自动停售
10. **Given** 用户访问微信小程序, **When** 浏览产品, **Then** 小程序版本的产品列表/详情页适配小程序交互，共享核心业务逻辑

### User Story 3 - 预订下单 (Priority: P1)

消费者选择团期和人数后，填写出游人信息，确认订单并完成支付。支持从常用出游人列表快速填写，30分钟内未支付自动取消。

**Why this priority**: 预订下单是核心交易环节，直接产生收入。

**Independent Test**: 已登录且已实名的用户可以从产品详情页完成"选团期→填信息→确认→支付"全流程。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 用户选择团期和人数, **When** 提交预订请求, **Then** 系统实时计算总价（含单房差自动附加：成人数为奇数时自动附加1份），锁定库存，返回订单预览
2. **Given** 用户填写出游人信息, **When** 提交身份证号, **Then** 系统实时校验身份证号格式（校验码规则），不通过即时返回错误
3. **Given** 用户确认订单, **When** 提交订单, **Then** 系统创建订单（状态：待付款），启动30分钟支付倒计时，返回支付参数
4. **Given** 用户选择支付宝/微信支付, **When** 发起支付, **Then** 系统创建支付订单，返回支付参数供前端调起支付
5. **Given** 支付成功回调, **When** 系统收到支付渠道通知, **Then** 系统验签通过后更新订单状态为"待出行"，发送订单确认通知（短信+站内信）
6. **Given** 用户超过30分钟未支付, **When** 倒计时结束, **Then** 系统自动取消订单，释放库存，订单状态变为"已取消"
7. **Given** 儿童人数>0, **When** 提交出游人信息, **Then** 每位儿童必须关联至少一位成人；婴儿限制每成人最多携带1名

**前端页面维度：**

8. **Given** 用户在产品详情页点击"立即预订", **When** 进入预订流程, **Then** 进入四步向导页面，顶部展示步骤进度条（选团期→填出游人→附加服务→确认支付），底部展示实时价格汇总栏
9. **Given** 用户在第一步选择团期与人数, **When** 选择日期和人数, **Then** 展示团期日历选择器、成人/儿童/婴儿人数选择器（含最小最大值限制）、实时更新的总价（含单房差明细）、当前余位显示、成团人数提示
10. **Given** 用户在第二步填写出游人信息, **When** 填写表单, **Then** 展示每位出游人的姓名/身份证号/手机号/出生日期/性别输入表单，支持从常用出游人列表一键填充，身份证号实时校验，儿童需选择关联成人
11. **Given** 用户在第三步选择附加服务, **When** 勾选服务, **Then** 展示旅游保险/接送机等可选项（名称/原价/优惠价），勾选后实时更新总价
12. **Given** 用户在第四步确认订单, **When** 查看汇总, **Then** 展示产品信息、出游人列表、费用明细（产品价+单房差+附加服务=应付总额）、退改政策摘要，需勾选"已阅读并同意退改政策"方可提交
13. **Given** 用户进入支付页面, **When** 选择支付方式, **Then** 展示支付宝/微信支付选择、30分钟倒计时（分钟:秒格式），支付成功后跳转订单确认页
14. **Given** 用户在微信小程序预订, **When** 进入支付, **Then** 通过 wx.requestPayment 调起微信支付
15. **Given** 支付超时, **When** 倒计时归零, **Then** 页面提示"支付已超时，订单已自动取消"，引导用户重新下单

### User Story 4 - 订单管理 (Priority: P2)

消费者可以在个人中心查看所有订单，按状态筛选，查看订单详情，对符合条件的订单申请退款。

**Why this priority**: 订单管理是交易闭环的必要环节，但不影响首单产生。

**Independent Test**: 已有订单的用户可以查看订单列表、筛选状态、查看详情、提交退款申请。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 用户请求订单列表, **When** 传入状态筛选参数, **Then** 返回该用户的订单列表（支持全部/待付款/待出行/退款中/已完成/已取消六种状态筛选），按下单时间倒序排列
2. **Given** 用户请求订单详情, **When** 传入订单 ID, **Then** 返回订单完整信息（产品信息、出游人列表、费用明细、支付记录、退改政策原文）
3. **Given** 用户提交退款申请, **When** 选择退款原因, **Then** 系统根据退改规则自动计算可退金额，创建退款申请进入审核流程
4. **Given** 退款审核通过, **When** 系统执行退款, **Then** 调用支付渠道退款接口，款项原路退回用户支付账户
5. **Given** 订单状态变更, **When** 系统处理状态流转, **Then** 按照订单状态机（待付款→待出行→出行中→已完成；待付款→已取消；待出行/待付款→退款中→已退款）正确流转

**前端页面维度：**

6. **Given** 用户进入个人中心"我的订单", **When** 打开页面, **Then** 展示顶部状态 Tab（全部/待付款/待出行/退款中/已完成/已取消）和订单卡片列表，每张卡片展示产品主图、名称、出发日期、金额、状态标签和可操作按钮
7. **Given** 用户查看订单详情, **When** 浏览详情页, **Then** 展示产品信息（含行程概要）、出游人信息列表、费用明细、支付记录、退改政策原文，不同状态展示不同操作按钮（待付款：立即支付+取消订单；待出行：申请退款；已完成：去评价）
8. **Given** 用户申请退款, **When** 点击"申请退款", **Then** 展示退款申请表单（退款原因选择、说明输入、退款金额预览（按退改规则自动计算）），提交后显示"退款申请已提交"提示
9. **Given** 用户在微信小程序查看订单, **When** 浏览订单列表, **Then** 小程序版本适配移动端交互，功能与 Web 端一致

### User Story 5 - 后台产品管理 (Priority: P2)

供应商可以发布境内游产品（基础信息、行程、价格日历、团期、库存），运营人员审核产品后上架。运营人员可以管理价格日历和库存。

**Why this priority**: 产品供给是平台运转的基础，但可以由平台运营人员代录，不依赖供应商自助操作。

**Independent Test**: 供应商可以创建产品并提交审核，运营人员可以审核通过/驳回，审核通过后产品在C端可见。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 供应商创建产品, **When** 提交基础信息, **Then** 系统校验必填项（产品名称/目的地/天数/出发地/交通方式/成团设置/费用说明），生成产品编号（DOM-{品类码}-{YYYYMMDD}-{序号}），产品状态设为"草稿"
2. **Given** 供应商编辑行程, **When** 为每天添加景点/用餐/住宿信息, **Then** 系统按天数生成行程卡片，支持行程模板保存与复用
3. **Given** 供应商配置价格日历, **When** 设置团期和价格, **Then** 支持按日设置成人价/儿童价/婴儿价/单房差价/库存数量，支持批量调价（固定价格/百分比调整/固定金额调整）
4. **Given** 供应商提交产品审核, **When** 点击"提交审核", **Then** 产品状态变为"待审核"
5. **Given** 运营人员审核产品, **When** 点击"通过", **Then** 产品状态变为"已上架"，C端可见；点击"驳回"需填写驳回原因
6. **Given** 已上架产品关键字段变更, **When** 供应商修改价格/行程/天数/退改规则, **Then** 产品状态变为"变更待审"，需重新审核
7. **Given** 运营人员管理库存, **When** 查看团期余位, **Then** 系统展示实时余位（总库存-已确认订单-待支付订单-留位+已取消释放），支持手动调整库存

**前端页面维度：**

8. **Given** 供应商登录后台, **When** 打开产品管理页, **Then** 展示产品列表（支持按状态/名称/目的地筛选），点击"发布产品"进入多步骤表单（基础信息→行程编辑→价格配置→退改规则→库存设置）
9. **Given** 供应商编辑行程, **When** 使用行程编辑器, **Then** 展示按天数自动生成的行程卡片，每张卡片包含标题/行程概述/景点选择/用餐安排/住宿酒店/交通方式/行程图片，支持拖拽排序
10. **Given** 供应商配置价格日历, **When** 打开价格日历页, **Then** 展示月历网格视图，每格显示成人价/库存状态/特殊标记，支持点击直接编辑和批量调价（五种模式：固定价格/百分比/固定金额/公式/跟随）
11. **Given** 运营人员打开产品审核页, **When** 查看待审核列表, **Then** 展示待审核产品列表，点击查看产品完整信息，操作按钮：通过（需确认）/驳回（需填写原因）
12. **Given** 运营人员管理退改规则, **When** 打开退改规则配置页, **Then** 展示阶梯费率编辑器（按距出发天数设置退款比例），支持保存为模板复用

### User Story 6 - 后台订单管理 (Priority: P2)

运营人员可以查询和处理订单，审核退款申请，配置退改规则（阶梯费率）。

**Why this priority**: 后台订单处理是交易闭环的运营支撑，但可以先用简化流程上线。

**Independent Test**: 运营人员可以搜索订单、查看详情、审核退款申请、配置退改规则模板。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 运营人员查询订单, **When** 传入筛选条件（订单号/手机号/状态/日期范围/产品类型/供应商）, **Then** 返回匹配的订单列表，支持分页和排序
2. **Given** 运营人员查看订单详情, **When** 传入订单 ID, **Then** 返回订单完整信息（产品/出游人/支付/操作日志）
3. **Given** 运营人员审核退款, **When** 查看退款申请, **Then** 系统展示按退改规则计算的应退金额、已发生费用、实际退款金额
4. **Given** 退款金额 ≤1000元, **When** 提交审批, **Then** 运营人员可直接审批；1000-5000元需财务主管审批；>5000元需总监审批
5. **Given** 退款审批通过, **When** 执行退款, **Then** 系统调用支付渠道退款接口，原路退回

**前端页面维度：**

6. **Given** 运营人员打开订单管理页, **When** 查看订单列表, **Then** 展示订单列表表格（订单号/产品名称/客户/金额/状态/下单时间），支持多维度筛选（订单号/手机号/状态/日期范围/产品类型）和导出
7. **Given** 运营人员查看订单详情, **When** 浏览详情页, **Then** 展示完整订单信息（产品/出游人/费用明细/支付记录/操作日志），操作按钮根据状态动态展示
8. **Given** 运营人员审核退款申请, **When** 打开退款审核页, **Then** 展示退款明细（应退金额/已发生费用/实际退款）、退改规则匹配结果、审批操作按钮（通过/驳回需填写原因），金额超过审批阈值时提示需上级审批

### User Story 7 - 基础权限管理 (Priority: P3)

系统管理员可以创建和管理运营人员、供应商账号，配置角色和权限（RBAC），确保不同角色只能访问授权功能。

**Why this priority**: 权限管理是安全保障，但MVP阶段角色较少，可以用简化方案启动。

**Independent Test**: 管理员可以创建账号、分配角色，不同角色登录后看到不同的菜单和功能。

**Acceptance Scenarios**:

**后端 API 维度：**

1. **Given** 管理员创建账号, **When** 提交用户信息和角色, **Then** 系统创建账号并生成初始密码（首次登录强制修改）
2. **Given** 管理员配置角色, **When** 为角色分配菜单/按钮/API 权限, **Then** 系统保存权限配置
3. **Given** 用户登录, **When** 系统验证身份, **Then** 返回该用户的角色和权限列表，前端据此动态渲染菜单
4. **Given** 供应商登录, **When** 查询产品/订单, **Then** 系统强制过滤为仅该供应商的数据（数据权限隔离）
5. **Given** 用户尝试访问未授权功能, **When** API 校验权限, **Then** 返回 403 并记录审计日志

**前端页面维度：**

6. **Given** 管理员打开用户管理页, **When** 查看用户列表, **Then** 展示运营人员/供应商账号列表（用户名/手机号/角色/状态），支持创建新账号（填写信息+选择角色）、冻结/解冻操作
7. **Given** 管理员打开权限管理页, **When** 配置角色, **Then** 展示角色列表和权限树（菜单权限+按钮权限），支持为角色勾选权限项，树形结构展示菜单层级
8. **Given** 不同角色登录后台, **When** 查看菜单, **Then** 仅展示该角色授权的菜单项，未授权菜单不可见，未授权按钮置灰或隐藏
9. **Given** 管理员执行敏感操作（退款审批>1000元、权限变更、数据导出）, **When** 点击执行, **Then** 系统弹出 MFA 验证弹窗，要求输入 TOTP 动态码或短信验证码，验证通过后方可执行

### Edge Cases

- **并发下单**：多人同时预订同一团期的最后几个名额时，系统如何处理？假设采用库存预扣机制，先到先得，超时未支付自动释放。
- **支付回调延迟**：支付渠道回调延迟或丢失时，系统应主动查询支付状态，避免订单状态不一致。
- **退款到账户失败**：退款执行后用户账户未到账（如银行卡异常），系统应记录异常状态并通知运营人员人工处理。
- **实名认证失败**：公安库校验不通过时，允许用户重新提交，但限制每日尝试次数（3次/天）。
- **微信登录已绑定手机号**：微信授权登录时若 OpenID 已绑定手机号，直接登录；若未绑定，引导用户绑定已有手机号或创建新账号。
- **产品审核驳回后重新提交**：供应商修改产品信息后可重新提交审核，系统记录审核历史。
- **单房差计算边界**：成人数为0（仅儿童/婴儿预订）时不计算单房差；成人数为1时附加1份单房差。
- **支付超时后用户仍完成支付**：超时取消后收到支付成功回调，系统应拒绝确认并触发退款流程。
- **身份证号校验**：系统对身份证号执行18位校验码规则验证（ISO 7064:1983.MOD 11-2），不通过即时提示。
- **团期售罄后用户仍在预订流程中**：用户在填写出游人信息期间团期售罄，确认订单时系统应提示"该团期已售罄"并引导选择其他日期。

## Requirements

### Functional Requirements

**用户模块**

- **FR-001**: 系统 MUST 支持手机号+短信验证码注册，验证码为6位数字，有效期5分钟，同一手机号60秒内不可重复发送
- **FR-002**: 系统 MUST 支持微信授权登录（OAuth 2.0），首次登录自动创建平台账号并绑定 OpenID
- **FR-003**: 系统 MUST 支持实名认证，校验姓名+身份证号的真实性，认证通过后可下单境内游产品
- **FR-004**: 系统 MUST 支持常用出游人管理，用户可预录入最多20位出游人的姓名、身份证号、手机号等信息
- **FR-005**: 系统 MUST 实施密码策略：8位以上，包含大小写字母、数字、特殊字符中的至少三种组合，密码有效期90天（到期前14天提醒更换，过期后强制修改）
- **FR-006**: 系统 MUST 在连续5次登录失败后锁定账号15分钟

**产品模块**

- **FR-007**: 系统 MUST 支持境内游产品的完整发布流程：基础信息→行程编辑→价格配置→退改规则→库存设置
- **FR-008**: 系统 MUST 支持产品审核流程：供应商提交→运营审核（通过/驳回）→上架/退回修改，已上架产品关键字段变更 MUST 触发重新审核
- **FR-009**: 系统 MUST 支持团期日历管理，按日设置成人价、儿童价、婴儿价、单房差价和库存数量
- **FR-010**: 系统 MUST 支持产品列表筛选：目的地、出发城市、行程天数、价格区间、出发日期
- **FR-011**: 系统 MUST 支持产品列表排序：综合推荐、价格升降序、满意度、销量
- **FR-012**: 系统 MUST 在产品详情页展示：图片轮播、行程详情（按天）、费用包含/不含、退改政策（不可折叠隐藏）、团期日历（日历热力图）、用户评价（5星制+各维度评分）
- **FR-013**: 系统 MUST 支持批量调价（固定价格/百分比/固定金额/公式/跟随五种模式）和节假日价格模板

**订单模块**

- **FR-014**: 系统 MUST 支持四步预订流程：选团期人数→填出游人信息→选择附加服务→确认订单与支付
- **FR-015**: 系统 MUST 在成人数为奇数时自动附加单房差费用，并展示明细
- **FR-016**: 系统 MUST 在订单创建后启动30分钟支付倒计时，超时自动取消并释放库存
- **FR-017**: 系统 MUST 支持订单状态流转（参照 PRD 表6-5），内部使用 9 个状态（pending_pay/paid_full/pending_travel/in_travel/completed/cancelled/refunding/refunded/closed），C端用户可见 6 个状态 Tab（全部/待付款/待出行/退款中/已完成/已取消），其中"待出行"合并展示 paid_full 和 pending_travel 两个内部状态。状态值统一使用 snake_case 小写格式
- **FR-018**: 系统 MUST 支持退款申请，根据退改规则自动计算应退金额，进入审核流程
- **FR-019**: 退款 MUST 原路退回用户原支付账户
- **FR-020**: 系统 MUST 支持退改规则配置，按距出发日期的天数阶梯设置退款比例（参照 PRD 表6-6）
- **FR-021**: 系统 MUST 支持退款分级审批：≤1000元运营审批、1000-5000元财务主管审批、>5000元总监审批

**支付模块**

- **FR-022**: 系统 MUST 支持支付宝支付（PC网站支付+手机网站支付）
- **FR-023**: 系统 MUST 支持微信支付（Native扫码支付+JSAPI支付+小程序支付）
- **FR-024**: 系统 MUST 处理支付回调的幂等性，同一笔支付的重复回调 MUST 不产生重复订单
- **FR-025**: 系统 MUST 在支付渠道回调丢失时主动查询支付状态

**后台管理模块**

- **FR-026**: 系统 MUST 支持 RBAC 权限模型，控制菜单可见性和按钮操作级权限
- **FR-027**: 系统 MUST 支持供应商数据隔离，供应商仅能看到自己发布的产品和相关订单
- **FR-028**: 系统 MUST 记录所有关键操作的审计日志（登录、产品审核、退款审批、权限变更），留存≥6个月
- **FR-030**: 系统 MUST 对管理员敏感操作（退款审批、权限变更、数据导出）要求 MFA 多因素认证（TOTP 动态码 + 短信验证码）

**附加服务**

- **FR-029**: 系统 MUST 支持附加服务（旅游保险、接送机）作为订单的可选项，用户可在预订流程中勾选，附加服务价格计入订单总价

### Key Entities

- **用户 (User)**: 消费者账号，包含手机号、昵称、头像、实名认证状态、会员等级
- **出游人 (Traveller)**: 出行人员信息，包含姓名、身份证号（加密存储）、手机号、出生日期、性别，关联到用户
- **产品 (Product)**: 境内游跟团游产品，包含名称、目的地、天数、行程、费用说明、退改规则、状态
- **团期 (Departure)**: 产品的可出发日期，包含日期、成人价、儿童价、库存数量、已售数量
- **订单 (Order)**: 消费者的预订记录，包含产品、团期、出游人、金额、状态、支付信息
- **退款申请 (Refund)**: 退款请求，关联订单，包含退款金额、原因、审核状态
- **附加服务 (Addon)**: 保险、接送机等可选服务，关联到订单，包含类型、价格
- **角色 (Role)**: RBAC 角色，包含角色名称、权限列表
- **退改规则 (CancellationRule)**: 按距出发天数阶梯定义退款比例，关联到产品

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户可以在3分钟内完成从注册到首单支付的全流程
- **SC-002**: 产品列表页加载时间不超过1秒（P95），首屏渲染不超过1.5秒
- **SC-003**: 订单确认页响应时间不超过500毫秒（P99）
- **SC-004**: 系统支撑10,000名用户同时在线浏览和下单
- **SC-005**: 支付成功率不低于98%（排除用户主动取消）
- **SC-006**: 退款审核通过后，款项在3个工作日内退回用户账户
- **SC-007**: 系统可用性达到99.9%，月度停机时间不超过43分钟
- **SC-008**: MVP上线后30天内处理不少于300笔订单
- **SC-009**: 三端（Web/小程序/后台）所有页面均可正常访问和操作，无阻断性缺陷

## Assumptions

- **供应商由平台代为录入**：MVP阶段不开放供应商自助入驻，供应商账号由系统管理员创建，产品由供应商或运营人员代为录入
- **微信小程序为唯一移动端**：MVP阶段仅开发微信小程序，支付宝和抖音小程序在后续阶段扩展
- **全额支付为唯一支付模式**：MVP阶段不支持定金+尾款模式，所有订单需一次性全额支付
- **附加服务为简单商品**：旅游保险和接送机作为可勾选的附加项，MVP阶段不需要对接保险公司API，保险产品由供应商手动配置
- **评价系统为简化版本**：MVP阶段用户可在订单完成后提交评价（5星制+文字），评价自动发布无需审核，差评不触发自动预警（运营人员手动关注）
- **库存并发控制在规划阶段确定**：高并发场景下的库存扣减方案（Redis原子操作 vs 数据库行锁）在实施计划阶段确定，规格层面保证"不超售"的业务结果
- **小程序与Web共享账号体系**：微信小程序用户通过手机号绑定与Web端用户打通，首次小程序登录时若手机号已注册则自动关联
- **退改审批阈值按1000/5000元分级**：≤1000元运营审批、1000-5000元财务主管审批、>5000元总监审批
- **支付倒计时统一30分钟（MVP刻意决策）**：所有产品均使用30分钟支付倒计时，此决策覆盖PRD §5.2.1中"短线产品15分钟"的选项。MVP阶段不区分产品类型设置差异化倒计时，后续阶段如需支持可将倒计时时长配置为产品级参数
- **MVP不包含分销功能**：分销体系在一期（Phase 2）交付，MVP阶段不涉及
- **退改规则阶梯费率参照PRD表6-6**：实际配置按产品维度独立设置，表6-6为参考模板
- **订单状态机参照PRD表6-5**：9个核心状态的流转规则以PRD定义为准
